# 事实核查清单（Fact-Check Checklist）

## 第一遍：文档回溯核查

### 案例信息核查
- [ ] 案号是否与原文一致？（完整案号，含年份、法院简称、案件类型、编号）
- [ ] 当事人姓名是否与原文一致？（不得擅自将具体姓名改为"某某"）
- [ ] 法院名称是否与原文一致？
- [ ] 审理程序（一审/二审/再审）是否标注正确？
- [ ] 案由是否与原文一致？

### 事实信息核查
- [ ] 日期是否与原文一致？（签约日、起诉日、判决日等）
- [ ] 金额是否与原文一致？（成交价、赔偿额、诉讼请求额等）
- [ ] 事件经过是否与原文一致？（不得添加原文未提及的细节）
- [ ] 第三人/证人信息是否与原文一致？

### 法律适用核查
- [ ] 引用的法条编号是否与原文一致？
- [ ] 法院援引的法律依据是否与原文一致？
- [ ] 判决结果是否与原文一致？（主文内容、赔偿比例等）

---

## 第二遍：来源 / PKULaw 回源交叉验证

### 案例验证
- [ ] 案号是否能在用户提供材料、可用 PKULaw MCP 或网页来源中回源？
- [ ] 回源材料中的当事人是否与引用一致？
- [ ] 回源材料中的审结日期是否与引用一致？
- [ ] 回源材料中的法院是否与引用一致？
- [ ] 当前没有可用 PKULaw 权限时，是否已标 `[待补: 案例来源]`？

### 法规验证
- [ ] 法条编号是否能在用户提供材料、可用 PKULaw MCP 或网页来源中回源？
- [ ] 法条内容是否与引用一致？（逐字核对）
- [ ] 法条是否已被废止或修改？
- [ ] 引用的款项是否正确？
- [ ] 当前没有可用 PKULaw 权限时，是否已标 `[待补: 法条原文]`？

---

## 第三遍：逻辑一致性核查

### 内部一致性
- [ ] 案号与法院是否匹配？（如法院简称应能对应到同一法院）
- [ ] 当事人与案由是否匹配？
- [ ] 一审与二审判决是否逻辑衔接？
- [ ] 金额数字是否前后一致？（起诉额、判决额、赔偿比例计算）

### 时间线核查
- [ ] 事件发生顺序是否合理？
- [ ] 签约日 → 纠纷发生日 → 起诉日 → 判决日，顺序是否正确？
- [ ] 诉讼时效是否合理？

### 法律逻辑核查
- [ ] 法律适用与案件事实是否匹配？
- [ ] 构成要件是否完整？
- [ ] 判决结果与法律依据是否对应？

---

## 常见事实错误类型

| 错误类型 | 示例 | 防范措施 |
|----------|------|----------|
| 案号错误 | (2011)写成(2012) | 逐字核对原文 |
| 金额错误 | [金额A]写成[金额B] | 逐位核对 |
| 当事人错误 | 王某写成汪某 | 核对原文姓名 |
| 法条错引 | 第148条写成第147条 | 用用户来源或 PKULaw 回源核验 |
| 法条意译 | 将法条内容改写后当原文引用 | 核对原文，不要凭记忆改写 |
| 编造细节 | 原文未提及但自行补充 | 严格回溯文档 |
| 判决结果错误 | [比例A]写成[比例B] | 核对原文判决主文 |
| 时间线错乱 | 签约日在纠纷之后 | 梳理时间线 |
