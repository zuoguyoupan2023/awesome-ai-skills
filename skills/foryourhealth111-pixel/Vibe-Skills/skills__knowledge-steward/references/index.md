# Knowledge Steward - 知识管理哲学与使用指南

## 核心哲学：从"流"到"库"

### 苏格拉底之问

> 河水流过没有任何痕迹，但如果修了水库，水就变成了势能。这套系统的本质是什么？

### 马克思主义视角

Knowledge Steward是**"劳动力的固化装置"**。它把你稍纵即逝的灵感（可变资本）转化为永久存储的文档（固定资本），供你在未来无限次复用。

## 解决的三大问题

### 1. 记忆磨损
**问题**：当时觉得很深刻，睡一觉全忘了。

**解决**：零摩擦记录。在Claude Code对话中直接保存，无需切换窗口、打开Obsidian或手动排版。

### 2. 知识碎片化
**问题**：经验散落在聊天记录、记事本和脑子里，无法系统化检索。

**解决**：结构化存储。统一的YAML元数据、自动分类、标签系统，让知识可搜索、可关联。

### 3. 复利阻断
**问题**：每次遇到相同问题都要重新思考一遍，无法积累经验。

**解决**：知识复利。建立个人经验库，下次遇到类似问题时可以快速检索历史解决方案。

## 使用场景详解

### 场景A：提示词复盘

**触发时机**：
- 发现某个Prompt让Claude的回答质量显著提升
- 某个Prompt完全没效果，需要记录避坑
- 总结出一套有效的提示词模板

**示例**：
```
用户："Claude，把你刚才用的那个关于代码重构的Prompt保存下来，标记为'高效'。"

Claude会：
1. 提取最近使用的提示词
2. 分析为什么有效（利用了什么机制）
3. 生成标签：#提示词工程 #代码重构
4. 保存到 提示词/ 目录
```

**价值**：
- 建立个人提示词库
- 理解有效提示词的模式
- 避免重复试错

### 场景B：工程排坑

**触发时机**：
- 花了3小时解决一个看似愚蠢的Bug
- 遇到罕见的依赖冲突或配置问题
- 发现某个工具的非直观用法

**示例**：
```
用户："记录这个Bug：npm install时出现ERESOLVE依赖冲突，解决方法是使用--legacy-peer-deps标志。"

Claude会：
1. 记录问题描述和解决方案
2. 分析根本原因（npm 7+的严格peer依赖检查）
3. 生成标签：#npm #依赖管理 #问题修复
4. 保存到 问题修复/ 目录
```

**价值**：
- 避免重复踩坑
- 建立个人问题库
- 理解问题的本质原因

### 场景C：灵感闪现

**触发时机**：
- 突然想到一个架构优化的点子
- 在写代码时顿悟了一个设计模式的应用
- 发现一个可能的技术方案

**示例**：
```
用户："保存这个想法：使用事件溯源(Event Sourcing)来实现可审计的用户操作历史。"

Claude会：

```

**价值**：
- 捕获稍纵即逝的灵感
- 建立想法池，定期回顾
- 追踪想法的演化过程

### 场景D：模式总结

**触发时机**：
- 发现一个可复用的代码模式
- 总结出一套最佳实践
- 提炼出通用的解决方案

**示例**：
```
用户："保存这个模式：使用工厂模式+策略模式实现可扩展的支付网关。"

Claude会：
1. 记录模式描述和代码示例
2. 分析适用场景和优势
```

**价值**：

### 场景E：效率优化

**触发时机**：

## 工作流建议

### 每日捕获（Daily Capture）

**原则**：宁可多记，不要遗漏。

**方法**：
- 在Claude Code会话中随时使用触发词
- 不要打断思路，快速保存后继续工作
- 标题可以简短，内容可以粗糙，后续可以编辑

### 每周复盘（Weekly Review）

**时间**：每周五下午或周日晚上

**步骤**：
1. 在Obsidian中查看本周保存的所有笔记
2. 补充分析和个人见解
3. 将有价值的笔记status改为`processed`
4. 删除或归档价值不大的笔记

**Dataview查询**：
```dataview
TABLE type, tags
FROM "Claude_Insights"
WHERE file.cday >= date(today) - dur(7 days)
SORT file.ctime DESC
```

### 每月归档（Monthly Archive）

**时间**：每月最后一天

**步骤**：
1. 回顾本月所有processed笔记
2. 提炼高价值笔记，形成知识卡片
3. 识别重复出现的模式和主题
4. 更新个人最佳实践文档

### 季度回顾（Quarterly Retrospective）

**时间**：每季度末

**步骤**：
1. 分析三个月积累的所有笔记
2. 识别知识盲区和成长方向
3. 总结个人工作模式的演变
4. 制定下季度的学习计划

## 高级用法

### 1. RAG（检索增强生成）闭环

**未来愿景**：当你的笔记积累了100+条后，可以配置Claude Code读取这个目录。

**效果**：
```
用户："帮我写一个用户认证的代码"

Claude：
- 先检索你的笔记库
- 发现你之前保存过"JWT认证最佳实践"
- 基于你的历史经验生成代码
- "根据你上个月的记录，你更喜欢使用JWT而非Session，是否照旧？"
```

**这就是复利**：AI开始用你过去的智慧来增强现在的你。

### 2. 标签系统优化

**原则**：
- 使用层级标签：`#编程/Python/异步编程`
- 定期合并同义标签
- 建立个人标签词典

**示例标签体系**：
```
#提示词工程
  ├─ #提示词工程/苏格拉底式
  ├─ #提示词工程/思维链
  └─ #提示词工程/角色扮演

#架构设计
  ├─ #架构设计/微服务
  ├─ #架构设计/事件驱动
  └─ #架构设计/DDD

#问题修复
  ├─ #问题修复/依赖管理
  ├─ #问题修复/性能优化
  └─ #问题修复/安全漏洞
```

### 3. 与其他工具集成

**Obsidian插件推荐**：
- **Dataview**：强大的查询和统计
- **Calendar**：按日期查看笔记
- **Graph View**：可视化知识网络
- **Templater**：高级模板功能

**外部工具**：
- **Anki**：将重要笔记制作成记忆卡片
- **Notion**：同步到团队知识库
- **GitHub**：版本控制你的知识库

## 常见问题

### Q1: 我应该保存所有内容吗？

**A**: 不。只保存有价值的内容：
- ✅ 花了很长时间才搞懂的概念
- ✅ 效果显著的提示词或方法
- ✅ 罕见的问题和解决方案
- ✅ 突然的灵感和顿悟
- ❌ 常识性的内容
- ❌ 容易搜索到的信息
- ❌ 一次性的临时笔记

### Q2: 如何避免笔记堆积？

**A**: 定期复盘和归档：
- 每周复盘，删除价值不大的笔记
- 每月归档，提炼高价值内容
- 使用status字段标记处理状态
- 不要追求完美，重要的是持续积累

### Q3: 标签太多怎么办？

**A**: 建立标签词典：
- 创建一个`标签词典.md`文件
- 记录每个标签的定义和使用场景
- 定期合并同义标签
- 使用层级标签减少顶层标签数量

### Q4: 如何让笔记更有价值？

**A**: 补充个人见解：
- 系统生成的分析只是起点
- 在每周复盘时补充个人思考
- 记录"为什么"而不只是"是什么"
- 关联相关笔记，形成知识网络
