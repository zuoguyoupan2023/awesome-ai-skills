# 复习设计架构

本文件不是题型模板，也不是题型菜单。它只提供出题前的设计框架，帮助模型保持判断自由，同时避免明显错配。

## 核心判断

题型是表层形态，不是第一步决策。先完成六个判断，再自然生成题目：

1. 复习契约：用户要回忆笔记、理解概念、训练能力、模拟任务，还是诊断薄弱点。
2. 证据边界：哪些内容来自笔记原文，哪些是根据笔记改写，哪些是模型生成的模拟材料。
3. 任务载体：用户要操作的对象是什么，例如一段材料、一个情境、一组概念、一段输出要求或一个决策场景。
4. 认知动作：用户要做什么，例如识别、解释、比较、推断、生成、改写、应用、评估、纠错。
5. 交互方式：一次性题单、逐题作答、选择控件、开放回答、答案版或保存为复习记录。
6. 评价依据：答完后凭什么批改，依据是笔记证据、题目材料、评分标准、逻辑一致性，还是用户给定目标。

如果这六项里有关键项缺失，不要用模板补洞；先请求补充，或明确降级为有限练习。

## 匹配原则

- 同一条信息可以在不同契约下有不同价值。考试范围、分值、题源在“背笔记”契约下可能是好题；在“能力训练”契约下通常只是设计线索。
- 真实任务练习需要任务载体。没有可操作的材料或情境，就不要把题目标成真实能力题。
- 模拟材料可以生成，但必须说明它是模拟、改写或基于笔记生成，不要伪装成笔记原文。
- 题目应让用户产生可批改的行为。只让用户确认自己看过目录、记住来源，通常只能算范围回忆。
- 题量、题型、难度都服务于契约，不服务于凑齐某个固定配比。

## 输出前闸门

输出前只检查这些宏观问题：

- 契约一致：题目标签、题干、材料和用户目标是否一致。
- 证据清楚：原文、改写、模拟是否区分清楚。
- 任务自洽：题目规则、选项、空格、材料、答案空间是否互相矛盾。
- 可批改：用户答完后是否能根据笔记或题目材料给出具体反馈。
- 不泄题：自测模式是否隐藏答案；答案版是否由用户明确要求。
- 交互合适：如果进入逐题选择作答，用 `ask_user` 收集选择；如果只是整套试卷预览，不要让交互控件破坏题卷结构。
